# GameStudio-Lite v0.1.0 改造实施方案
## 1. 目标
本次改造目标是在不破坏原 `Claude-Code-Game-Studios` 架构的前提下，基于独立开发者的实际使用场景，对原完整游戏工作室框架进行轻量化调整，衍生出 `GameStudio-Lite v0.1.0` 版本。
`GameStudio-Lite` 的核心定位是：
> 面向独立开发者的 Claude Code 游戏开发轻量工作流框架。
它主要服务两类场景：
1. 快速验证 Demo：用于快速完成玩法想法验证、核心循环验证、Vertical Slice 或短周期可玩版本。
2. 单个项目开发：用于支持独立开发者完成一个相对完整的游戏项目，从概念、设计、开发、测试、打磨到提交。
`GameStudio-Lite` 不是对原项目的推翻重写，而是在原 `Claude-Code-Game-Studios` 的 agents、skills、workflow、hooks、templates 基础上，新增一层更适合独立开发者的轻量化使用方式。
本版本优先解决以下问题：
1. 原框架默认偏完整工作室流程，对独立开发者来说过重。
2. 原框架 agents、skills、templates 数量较多，初次使用成本较高。
3. 对快速 Demo 和单个项目开发来说，需要更明确的默认路径。
4. 需要减少不必要的文档、Agent 调用和 token 消耗。
5. 需要保留原框架完整能力，同时提供更轻量的默认工作流。
后续版本会在 `v0.1.0` 基础上继续迭代，包括但不限于：
1. 更完善的 Skill Metadata。
2. 更稳定的项目状态管理。
3. 更精细的模板 brief / full 落地。
4. 更便捷的框架同步方式。
5. 更适合具体游戏项目的初始化流程。
6. 更完善的 Claude Code 使用验收流程。
---
## 2. 改造范围
`v0.1.0` 必须完成以下内容：
1. 新增 `Project Mode` 项目模式说明，支持：
   - `Jam Mode`：快速交付 / Demo 验证模式。
   - `Indie Mode`：完整独立游戏项目模式。
2. 新增 `Agent Packs` 角色包说明，用于在不删除原 49 个 agents 的前提下，按项目阶段组合使用 agents。
3. 新增 `Skill Catalog Lite` 技能分类说明，将 skills 分为：
   - `Core`
   - `Optional`
   - `Archive`
4. 新增 `Skill Metadata` 轻量元数据标准，用于描述 Core Skills 的阶段、模式、风险和下一步建议。
5. 新增 `Template Mode` 模板模式说明，支持：
   - `brief`
   - `full`
6. 新增 `project_state.yaml.example`，作为游戏项目使用 Lite 框架时的项目状态模板。
7. 新增 `lean-check` command，用于检查当前计划、系统设计、Agent 使用、Skill 使用、Template 使用是否过度工程化。
8. 更新 `CLAUDE.md`，补充 Lite 版本的全局使用规则。
9. 更新 `README.md`，说明 Lite 版本定位、使用方式和同步到游戏项目的方法。
10. 新增或更新 `CHANGELOG.md`，记录 `v0.1.0` 改造内容。
---
## 3. 非目标
`v0.1.0` 明确不做以下内容：
1. 不删除、重命名或重写原有 agents。
2. 不取消原项目的三层工作室结构。
3. 不重写全部 skills。
4. 不重写原 workflow catalog。
5. 不修改 hooks 或 hook 行为。
6. 不修改 `.claude/settings.json`。
7. 不新增复杂 `Agent Router` 系统。
8. 不新增 Git submodule 支持。
9. 不将项目改造成 Codex 专属架构。
10. 不替换 Claude Code slash commands。
11. 不创建具体游戏项目代码。
12. 不新增复杂同步脚本作为 v0.1.0 必须项。
13. 不把 `project_state.yaml.example` 扩展成大型项目管理系统。
14. 不让 `lean-check` 自动修改文件。
---
## 4. 设计规则
Codex 执行修改时必须遵守以下规则：
1. 保留原 Claude Code 架构。
2. 保留原 49 个 agents，作为底层角色库。
3. 保留原三层工作室结构：
   - Directors 总监层。
   - Department Leads 部门负责人层。
   - Specialists 专家执行层。
4. 新增轻量化使用层，而不是替换原结构。
5. 使用 `Agent Packs` 作为默认使用入口。
6. 使用 `Project Mode` 决定默认流程强度。
7. `Jam Mode` 默认使用 brief 模板。
8. `Indie Mode` 默认使用 brief，核心高风险系统可使用 full。
9. `lean-check` 是按需运行的 command，不是常驻监督 agent。
10. 优先新增文档和配置，不做大规模重构。
11. 所有改动必须保持 Claude Code 可读、可执行、可维护。
12. 每批修改后必须输出修改文件列表和修改摘要。

6. 模块需求
6.1 Project Mode：Jam / Indie

新增文件：
.claude/docs/project-modes.md

该文件必须定义两种项目模式。
6.1.1 Jam Mode

Jam Mode 是快速交付 / Demo 验证模式。

适用场景：

1. 游戏比赛。
2. Game Jam。
3. 快速玩法 Demo。
4. Vertical Slice。
5. 短周期可玩版本。
6. 以验证核心体验和完成可提交版本为目标的项目。

默认行为：

project_mode: jam
template_mode: brief
review_intensity: lean
default_skill_set: jam_core
max_default_agents: 5
primary_goal: playable_first
target_playtime_minutes: 30

优先级：

可玩闭环 > 玩家能理解 > 体验完整 > 提交可用 > 架构优雅

默认不启用：

localization
live-ops
community
monetization
full economy modeling
heavy networking
large-scale analytics

⸻

6.1.2 Indie Mode

Indie Mode 是完整独立游戏项目模式。

适用场景：

1. 独立游戏长期开发。
2. 单个完整游戏项目。
3. Steam Demo。
4. 小团队项目。
5. 可维护、可扩展的游戏项目。
6. 需要多个里程碑和持续迭代的项目。

默认行为：

project_mode: indie
template_mode: brief_by_default
review_intensity: standard
default_skill_set: indie_core
max_default_agents: 8
primary_goal: sustainable_iteration

使用规则：

1. 默认使用 brief 模板。
2. 核心高风险系统可使用 full 模板。
3. 跨系统改动需要更强 review。
4. 可启用更多 Agent Pack，但仍不默认启用全部 49 agents。

6.4 Skill Metadata：技能元数据

skill-catalog-lite.md 中需要定义 Core Skill 的轻量元数据标准。

字段控制在 7 个以内：

id:
stage:
modes:
category:
agent_pack:
risk:
next:

字段说明：

id：skill 命令名称
stage：适用阶段
modes：适用项目模式
category：skill 类型
agent_pack：默认使用的 Agent Pack
risk：执行风险，low / medium / high
next：推荐下一步

示例：

id: dev-story
stage: production
modes: [jam, indie]
category: execution
agent_pack: production
risk: high
next: [story-done, code-review]

再示例：

id: lean-check
stage: any
modes: [jam, indie]
category: review
agent_pack: core
risk: low
next: [scope-check, quick-design, dev-story]

注意：

v0.1.0 只要求 Core Skills 有轻量 metadata。
不要为所有 skills 强制补全复杂 metadata。

6.7 lean-check

新增文件：

.claude/commands/lean-check.md

如果原项目 commands 目录路径不同，Codex 需要根据实际仓库结构选择最接近的 Claude Code slash command 位置。

lean-check 定义：

lean-check 是反过度工程化检查 command。
它只输出检查结果，不默认修改文件。

必须检查以下维度：

1. Scope：当前功能范围是否过大。
2. Agents：当前是否调用过多 agents。
3. Skills：当前 skill 是否适合当前模式。
4. Templates：当前是否应使用 brief 而不是 full。
5. Technical Complexity：当前技术方案是否过早复杂化。
6. Player Value：当前改动是否直接提升玩家体验。
7. Delivery Risk：当前计划是否影响可交付性。

输出格式：

# Lean Check Result
## Current Mode
## Verdict
Keep / Simplify / Cut / Defer
## Findings
### Scope
### Agents
### Skills
### Templates
### Technical Complexity
### Player Value
### Delivery Risk
## Recommendation
### Keep
### Simplify
### Cut
### Defer
## Next Step

运行规则：

1. 先读取 project_state.yaml 或 project_state.yaml.example。
2. 判断当前 mode 是 jam 还是 indie。
3. 不修改文件。
4. 不新增任务。
5. 只给出收敛建议。
6. Jam Mode 中优先保护可玩闭环、Demo 验证和提交可用性。
7. Indie Mode 中优先保护长期可维护性，但仍避免过早复杂化。

10. 风险与防误改约束

⸻

10.1 主要风险
1. Codex 过度重构原项目。
2. Codex 修改了 hooks 或 settings。
3. Codex 删除或重写原 agents。
4. Codex 将 Lite 改造成 Codex 专属结构。
5. Codex 一次性修改过多文件，难以回滚。
6. lean-check 被实现成自动改文件的 agent。
7. project_state.yaml.example 变得过于复杂。
8. README 被改得过长，影响原项目信息结构。

⸻

10.2 防误改约束

Codex 必须遵守：
1. 不删除、重命名、重写原 agents。
2. 不修改 hooks。
3. 不修改 .claude/settings.json。
4. 不重写 workflow catalog。
5. 不大规模重写 existing skills。
6. 不新增复杂 Agent Router。
7. 不新增 sync-to-project.sh 作为 v0.1.0 必须功能。
8. 不创建具体游戏项目代码。
9. 不让 lean-check 自动修改文件。
10. 不把 project_state.yaml.example 扩展成完整项目管理系统。
11. 不将 Claude Code slash commands 改为 Codex-only workflows。
12. 每批修改必须单独总结。

⸻

10.3 回滚方案
1. 每个 Batch 单独 commit。
2. 如果某一批出问题，只回滚该批 commit。
3. 不在 main 分支直接修改。
4. 所有改动先进入 dev。
5. main 只合入已验收内容。

回滚命令示例：

git revert <commit_hash>
